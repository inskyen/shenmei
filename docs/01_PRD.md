# 01_PRD：審美者產品需求文檔

## 1. 產品定位

**審美者** 是一個以影片為入口的審美採樣社交產品。

使用者像發 Soul 瞬間動態一樣發布一條內容，但這條內容不是純心情或自拍，而是：

```text
一支影片 + 一段推薦理由 + 可選頻道
```

審美者不做普通影片站，也不做單純收藏夾。它真正展示的是：

> 某個人為什麼選擇這支影片，以及這個選擇透露出他的審美。

## 2. 核心判斷

### 2.1 video + post 雙核心

產品採用 `video` + `post` 雙核心。

- `video`：影片本體資料，是內容沉澱核心，像 B 站影片頁。
- `post`：某個使用者對某支影片的一次採樣推薦，是信息流傳播與個人審美表達核心，像 Soul 瞬間動態。

同一支影片可以被多人重複推薦。每一次推薦都因為推薦者、推薦理由、頻道歸屬不同，而成為不同的內容。

頁面分工：

```text
/v/[id]  影片主頁：播放、影片資訊、所有推薦
/p/[id]  採樣動態詳情：某個人的推薦理由與圍繞該推薦的留言
```

### 2.2 推薦理由必填

推薦理由是產品靈魂。

如果只允許貼影片連結，審美者會退化成收藏夾。第一版應要求推薦理由必填，建議最少 10 個字。

### 2.3 第一版登入後即可發布

第一版不做邀請制，不做採樣人白名單。

規則：

- 使用者註冊 / 登入後即可發布。
- 發布內容需要通過基本表單校驗。
- 後續再考慮管理員精選、隱藏、檢舉與風控。

## 3. 目標使用者

第一批使用者是對內容質感有感的人：

- 喜歡收藏與分享 B 站影片、影像、音樂、動畫、設計內容的人。
- 不只想轉發連結，也願意寫幾句「為什麼推薦」的人。
- 想透過內容發現人、透過審美靠近人的人。

使用者心理：

- 我想留下自己的審美痕跡。
- 我想看到別人如何描述一支影片。
- 我想找到和我頻率相近的人。

## 4. 產品氣質

主氣質：

- 安靜展廳感
- 霧藍、柔白、低飽和
- 私人採樣
- 克制但有情緒
- 高級但不冷

暗線氣質：

- Soul 式輕社交
- 因內容而靠近
- 有追蹤、留言、私訊的可能

結構參考：

- 天涯 / 貼吧 / 豆瓣小組 / Moltbook 的板塊感
- 但不要做成重論壇，也不要過早加入等級、簽到、吧務等複雜系統。

## 5. MVP 範圍

第一版只做最小採樣閉環：

```text
註冊 / 登入
→ 發布 B 站影片
→ 填寫推薦理由
→ 出現在首頁最新流
→ 其他使用者觀看、喜歡、留言
→ 點進使用者頁或未來進一步私訊
```

### 5.1 MVP 必做

- 使用者註冊 / 登入。
- 使用者 profile 基礎資料。
- B 站影片提交。
- 影片本體資料保存。
- 採樣動態保存。
- 推薦理由必填。
- 首頁最新流展示採樣動態。
- 採樣動態詳情頁。
- 頻道列表與頻道頁基礎結構。
- 喜歡。
- 留言。

### 5.2 MVP 可先占位

- 追蹤流。
- 私訊入口。
- 通知入口。
- 頻道精選。
- 使用者個人頁中的完整資料。

### 5.3 第一版不做

- 多平台內容支持。
- 即時私訊。
- 推送通知。
- 演算法推薦。
- 複雜頻道治理。
- 完整收藏系統。
- 等級 / 積分 / 勳章。
- 大規模內容審核系統。

## 6. 核心功能模組

### 6.1 帳號與個人頁

作用：

- 讓每一條採樣動態有明確的推薦者。
- 支撐後續追蹤、私訊與個人審美檔案。

MVP：

- 註冊 / 登入。
- 暱稱。
- 頭像。
- 個人簡介。
- 個人發布列表。

後續：

- 審美標籤。
- 追蹤數 / 被追蹤數。
- 私訊權限。

### 6.2 影片本體

作用：

- 存放影片來源資料。
- 避免同一支影片被重複存多份。

MVP：

- BVID。
- 標題。
- 封面。
- UP 主。
- 原始連結。
- 來源平台。

### 6.3 採樣動態

作用：

- 產品核心內容。
- 表示某個使用者對某支影片的一次推薦。

MVP：

- 推薦者。
- 對應影片。
- 推薦理由。
- 發布時間。
- 可見性。
- 可選頻道。

### 6.4 首頁大廳

首頁是全站大廳，不屬於任何單一頻道。

頂部導航：

```text
追蹤 / 最新 / 頻道
```

第一版：

- `最新`：全站最新採樣動態。
- `頻道`：頻道入口。
- `追蹤`：顯示目前使用者追蹤之人的公開採樣動態。

底部導航：

```text
發現 / 頻道 / 採樣 / 私訊 / 我的
```

底部 `頻道` 與首頁頂部頁籤會切換至頻道列表。搜尋由首頁右上角入口進入 `/search`。

### 6.5 頻道

頻道是主題板塊，路由形態為 `/m/xxx`。

例子：

- `/m/blue-hour`：藍色時刻
- `/m/film`：影像
- `/m/late-night`：深夜放映
- `/m/lonely-summer`：孤獨夏天

第一版規則：

- 管理員建立頻道。
- 使用者發布時可選頻道。
- 發布時不強制選頻道，未選時預設發到大廳最新流。
- 一篇採樣動態可以屬於多個頻道。

### 6.6 留言與喜歡

MVP 先做：

- 喜歡。
- 留言。

資料模型預留兩類互動：

- 影片互動：圍繞 `video`，例如未來的影片頁公共評論。
- 動態互動：圍繞 `post`，例如對某個推薦理由的喜歡與留言。

MVP 只正式開通 `post` 的喜歡與留言。影片頁先作為安靜的檔案室，展示影片與所有推薦；當同一支影片累積足夠多推薦、出現跨推薦討論需求時，再開通 `video` 公共評論。

第一版 `reactions` 只做 `like`，不提前做多種表情反應。

刪除規則：

- 採樣作者可刪除自己的採樣，超管可刪除任何採樣；刪除採樣不刪除影片本體。
- 留言作者可刪除自己的留言，超管可刪除任何留言。
- 已有回覆的主留言刪除後顯示「此留言已刪除」，其他人的回覆繼續保留。

### 6.7 追蹤

第一階段可先建立資料結構，功能後續做。

追蹤對象：

- 使用者追蹤使用者。
- 後續可支持使用者追蹤頻道。

### 6.8 私訊

私訊是獨立核心模組，但不在第一版重開發。

定位：

```text
看到某人的採樣
→ 點進個人頁
→ 追蹤 / 留言
→ 進一步私訊
```

階段：

- 第一版：底部 `訊息` 入口占位。
- 第二版：文字私訊 + 未讀。
- 第三版：即時訊息 + 通知 + 封鎖 / 檢舉。

### 6.9 通知

通知與私訊分開建模，也應在路由上與私訊分開。

通知類型：

- 有人喜歡我的 post。
- 有人留言。
- 有人追蹤我。
- 有人私訊我。

未來路由：

```text
/messages       私訊
/notifications  通知
```

第一版可先預留，不必完整實作。

## 7. 核心使用流程

### 7.1 發布採樣

```text
使用者登入
→ 點擊發佈
→ 輸入 B 站連結 / BVID
→ 系統解析或手動填入影片資料
→ 使用者填寫推薦理由
→ 選擇頻道，可跳過
→ 發布成功
→ 回到大廳最新流，並立即看到自己剛發布的採樣動態
```

### 7.2 瀏覽最新流

```text
進入大廳
→ 查看最新採樣動態
→ 點擊影片封面進入放映室
→ 點擊文字 / 卡片進入詳情
→ 喜歡 / 留言 / 查看推薦者
```

### 7.3 頻道瀏覽

```text
點擊頻道
→ 進入頻道列表
→ 選擇某個頻道
→ 查看該頻道下的採樣動態
```

## 8. 成功標準

第一階段完成時，產品應達到：

- 使用者知道這不是普通影片站，而是採樣社交。
- 使用者能發布一支 B 站影片並寫推薦理由。
- 發布成功後會回到大廳最新流，讓使用者立即看到自己的內容被收錄。
- 首頁展示的是採樣動態，影片頁沉澱單支影片的完整內容。
- 同一支影片可以承載多個人的不同推薦。
- 頻道結構已存在，可承接主題內容。

## 9. 風險

### 9.1 定位風險

如果首頁只展示影片標題和封面，產品會變成影片列表。  
必須突出推薦者與推薦理由。

### 9.2 範圍風險

私訊、頻道治理、通知都很重要，但不能第一版全部重做。
第一版要先把採樣動態閉環打通。

### 9.3 資料結構風險

如果把推薦理由直接塞進 `videos`，後續會很難支持多人重複採樣。  
必須拆分 `videos` 和 `posts`，並保留影片頁 `/v/[id]` 作為長期沉澱入口。

## 10. 下一步

在正式寫功能代碼前，需要確認：

- 資料模型。
- 頁面路由。
- API / Supabase 讀寫契約。
- 開發階段拆分。
