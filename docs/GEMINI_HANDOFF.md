# Gemini 交接文檔：審美者目前狀態與前端原型修整建議

> 更新時間：2026-07-09  
> 目的：讓另一個 AI 或協作者快速理解「審美者」正在做什麼、目前做到哪裡、接下來可以怎麼修整前端原型。

## 1. 一句話理解產品

**審美者** 是一個以影片為入口的審美策展社交產品。

使用者發布的不是單純影片，而是：

```text
一支 B 站影片 + 一段推薦理由 + 可選小館
```

產品核心不是「熱門影片排行榜」，而是「某個人為什麼選擇這支影片」，讓影片成為審美、人格與社交的入口。

## 2. 產品核心結構

目前已確認採用 **video + post 雙核心**。

| 核心 | 含義 | 對應頁面 |
| --- | --- | --- |
| `video` | 影片本體，長期沉澱播放、所有推薦與公共討論 | `/v/[id]` |
| `post` | 使用者的一次推薦，進入信息流與個人頁 | `/p/[id]` |

首頁、大廳、小館、使用者頁主要展示 `posts`。  
影片頁 `/v/[id]` 才以 `video` 為中心展示所有推薦。

## 3. 目前技術棧

- Next.js Pages Router
- React
- Supabase Auth
- Supabase Database
- Vercel 部署已連過
- UI 目前以 inline style 為主，還不是正式組件化設計系統

- 未登入點受限功能會提示「請先登入」。

- 新增 `/settings` 設定頁與登出功能。

這批改動尚未 commit，修整前端原型時請不要覆蓋。

## 5. 已完成頁面與功能

### 5.1 大廳 `/`

目前已有：

- 最新策展流。
- 從 `/api/feed` 讀取公開已發布 posts。
- 卡片展示：
  - 策展人。
  - 發布時間。
  - 推薦理由。
  - 影片封面。
  - 影片標題。
  - 喜歡數。
  - 留言數占位。
- 點卡片進 `/p/[id]`。
- 點影片相關區域可進 `/v/[id]` 或沉浸播放。
- 點作者可進 `/u/[username]`。
- 點喜歡可寫入 / 刪除 reactions。
- 游客點喜歡會提示登入。

目前主要問題：

- UI 還偏原型，卡片資訊層級可以再整理。
- 首頁仍有部分中文註釋是簡體，不影響功能，但若要完整台灣字可再清理。
- 影片封面缺失時的空態還不夠好看。

### 5.2 登入 `/login`

目前已有：

- Email / Password 登入。
- Email / Password 註冊。
- 支援 `?next=` 回跳。
- 回跳路徑已限制為站內路徑。
- 已有較正式的產品入口視覺。

目前主要問題：

- 視覺還可以繼續精修，但已不是純工程表單。
- 還沒有 profile 編輯入口。

### 5.3 發布 `/submit`

目前已有：

- B 站連結 / BVID 輸入。
- 推薦理由必填。
- 推薦理由至少 10 個字。
- 可手動填影片標題、UP 主、封面 URL。
- 可選小館。
- 未登入提示登入。
- 已登入可建立：
  - `videos`
  - `posts`
  - `post_modules`
- 發布成功後回到 `/` 大廳最新流。

目前主要問題：

- 還不能自動解析 B 站影片標題 / 封面 / UP 主。
- 表單比較像後台工具，缺少社交產品的發布儀式感。
- 小館選擇第一版能用，但產品上還要再設計。

### 5.4 策展詳情 `/p/[id]`

目前已有：

- 展示單條 post。
- 展示 B 站播放器。
- 展示影片標題、UP 主、推薦理由、作者、日期。
- 展示小館標籤。
- 支援喜歡 / 取消喜歡。
- 游客點喜歡提示登入。
- 已有留言區靜態 UI。
- 游客可看到登入留言提示。
- 已登入可輸入留言草稿，但暫不保存到資料庫。

目前主要問題：

- 留言尚未接資料庫。
- 右上發佈入口已有登入守衛，但視覺仍粗。
- 作者區與影片區的信息層級可以再打磨。

### 5.5 影片主頁 `/v/[id]`

目前已有：

- 展示影片播放器。
- 展示影片標題與 UP 主。
- 統計推薦數與喜歡數。
- 展示這支影片下所有公開推薦。
- 可點推薦進 `/p/[id]`。
- 右上推薦按鈕會帶 BVID / title 到 `/submit`。
- 游客點推薦會提示登入。
- 已有影片公共留言區靜態 UI。
- 游客可看到登入留言提示。
- 已登入可輸入留言草稿，但暫不保存到資料庫。

目前主要問題：

- 影片留言尚未接資料庫。
- 還沒有相關小館展示。
- 影片頁可以再向「資料沉澱頁」靠攏，而不是只像列表頁。

### 5.6 搜尋 `/search`

目前已有：

- 搜尋已收錄影片。
- 支援影片標題、UP 主、BVID。
- 結果可進 `/v/[id]`。
- 結果可點推薦，帶資料進 `/submit`。
- 游客點推薦提示登入。

目前主要問題：

- 搜尋頁視覺還比較簡單。
- 搜不到時可以加「直接貼 BVID 發佈」的更完整引導。
- 未來要區分搜尋 video / 搜尋使用者 / 搜尋小館。

### 5.7 使用者頁 `/u/[username]`

目前已有：

- 展示 username、頭像、簡介占位。
- 展示策展數、收穫喜歡數。
- 展示該使用者發布過的公開 posts。
- `/u/me` 已登入時跳到自己的 username。
- `/u/me` 未登入時提示登入。

目前主要問題：

- 沒有編輯資料。
- 沒有關注 / 粉絲。
- 沒有私訊按鈕實作。
- 使用者頁可以更像「審美名片」。

### 5.8 小館 `/m`、`/m/[slug]`

目前已有：

- 小館列表。
- 小館詳情。
- 小館內策展列表。

產品決策補充：

- 使用者之前提到「小館比較複雜，要做得比較深度，先跳過」。
- 所以目前小館頁雖然能用，但不應作為下一個主要深挖點。

### 5.9 訊息 `/messages`

目前未做：

- 發送消息。
- 未讀狀態。

### 5.10 通知 `/notifications`

目前已有：

- 未登入訪問會提示登入。

目前未做：

- 未讀狀態。

### 5.11 設定 `/settings`

目前已有：

目前未做：

## 7. 游客 / 登入使用者規則

目前已實作的產品規則：

游客可以：

- 看首頁。
- 看影片頁。
- 看策展詳情。
- 看使用者頁。
- 搜尋影片。
- 瀏覽小館。

游客不能：

- 發佈策展。
- 喜歡。
- 看自己的頁面。
- 看訊息。
- 看通知。
- 從影片或搜尋結果推薦影片。

游客點受限功能時：

```text
先提示「請先登入」
再跳到 /login?next=原本想去的路徑
登入後回到原路徑
```

## 8. 目前最適合繼續做的功能池

下面按「適合現在做」排序。

### A. 前端原型修整：最推薦

目標：讓產品從「能用」變成「像一個有氣質的 App」。

可做內容：

- 首頁卡片重新排版。
- 封面缺失時的美觀占位。
- 底部導航 active 狀態。
- 頂部「追蹤 / 最新 / 小館」狀態更明確。
- 详情页、影片页、搜索页视觉统一。
- 登入頁已初步正式化，後續可再精修微互動。

難度：低到中。  
風險：低。  
是否需要改資料庫：否。

### B. 留言資料庫接入：稍微複雜，晚上適合做

`/p/[id]` 與 `/v/[id]` 都已有靜態留言 UI，下一步才是真正保存。

可做內容：

- 建立 comment insert / select 流程。
- 支援 `target_type = post` 的策展留言。
- 支援 `target_type = video` 的影片公共留言。
- 游客只能看，登入後才能留言。

難度：中。  
風險：中。  
是否需要改資料庫：可能需要確認 RLS policy。

### C. 使用者頁打磨：推薦

使用者頁是社交產品人格承載點，很值得做。

可做內容：

- 改成「審美名片」風格。
- 增加簡介、審美標籤展示。
- 增加「發佈策展」入口。
- 預留「私訊」與「追蹤」按鈕。
- 先不實作真私訊 / 真追蹤，只做狀態引導。

難度：低到中。  
風險：低。  
是否需要改資料庫：暫時否。

### D. 發布頁體驗優化：推薦

發布是 MVP 的核心閉環，值得打磨。

可做內容：

- 讓 BVID 輸入更像「貼上影片」。
- 封面 URL、標題、UP 主收進進階區。
- 增加「這支影片第一次被收錄」提示。
- 增加發布成功前後的文案。
- 小館選擇更像 tag chip。

難度：中。  
風險：低。  
是否需要改資料庫：否。

### E. 個人資料編輯入口：可以做

目前已有 `/settings` 和登出，但還不能改 profile。

可做內容：

- 在 `/settings` 增加 display name / bio / avatar_url 表單。
- 更新 `profiles` 表。
- 保留 username 修改到後面，避免唯一性處理變複雜。

難度：中。  
風險：中低。  
是否需要改資料庫：否，但需要確認現有 RLS policy 是否允許更新自己的 profile。

### G. 通知：稍後做

通知需要依賴喜歡、評論、回覆事件。

不建議現在先做真通知。  
可以先做通知頁 UI 原型。

### H. 私訊：後續大模組

私訊是 Soul 感的重要來源，但不是今天最適合的小功能。

原因：

- 要設計 conversations / messages。
- 要處理 RLS。
- 要處理未讀數。
- 要處理誰能私訊誰。

可以先做 UI 占位，不建議現在貿然接資料庫。

## 9. 給 Gemini 的前端修整方向

如果 Gemini 要幫忙修前端，建議先不要大改資料邏輯，優先做「視覺與交互一致性」。

### 9.1 建議先修首頁

首頁是產品第一印象。

建議方向：

- 卡片像「策展卡」而不是普通動態。
- 推薦理由要比影片標題更像主角。
- 影片封面是證據，不是唯一主角。
- 作者區可以更安靜。
- 喜歡 / 留言 / 分享底部操作要更輕。

### 9.2 登入頁需要正式化

現在登入頁是工程版。

建議方向：

- 標題：`登入審美者`
- 副標：`回到你的策展大廳`
- 如果從受限功能跳來，可以提示：

```text
登入後就能繼續剛剛的操作。
```

### 9.3 發布頁需要降低壓力

目前發布頁偏表單。

建議方向：

- 第一屏只問：
  - 貼上 B 站連結 / BVID
  - 為什麼推薦它？
- 影片標題、封面、UP 主作為「補充資訊」。
- 小館作為「放進哪個展間」。

### 9.4 詳情頁需要增加社交感

可先加靜態評論區：

```text
登入後一起聊聊這支影片。
```

或：

```text
還沒有人留言。留下第一個回聲。
```

## 10. 不建議 Gemini 現在做的事

暫時不要：

- 大改資料表。
- 重寫小館系統。
- 重寫整個首頁資料流。
- 把 inline style 一次性全部改成 CSS Modules。
- 做完整私訊。
- 做完整通知。
- 做複雜推薦算法。

原因：目前最重要的是保住 MVP 閉環，把前端體驗磨順。

## 13. 當前完成度判斷

以 MVP 角度：

```text
功能閉環：約 65%
資料鏈路：約 70%
前端完成度：約 45%
社交完成度：約 25%
產品文檔：約 75%
```

總體判斷：

> 審美者已經從純 UI 原型進入「可跑的策展社交 MVP 雛形」。  
> 下一階段不應急著擴大功能，而應把首頁、發布、詳情、登入、個人頁這幾個核心觸點磨順。
